# Backend Module Spec

本文件定义桌面智算 Python 后端边界。当前仓库后端位于 `backend/`，共享契约位于 `packages/shared/`。

## 服务边界

后端负责稳定业务能力和本机执行：

- 项目、任务、数据集、模型资产、运行环境、AI 模型配置的业务 API。
- 文件路径存在性、权限、大小和结构检查。
- 训练任务、推理服务、环境检测、依赖安装的受控执行。
- 状态持久化、日志采集、错误码和诊断数据。

AI 助手不能绕过后端直接改变稳定业务状态。

## API 设计原则

- API 返回结构必须稳定。
- 错误响应必须包含可用于 UI 和 AI 诊断的 `errorCode` 与 `errorMessage`。
- 本地路径、进程 ID、端口、日志引用等字段必须明确语义。
- 高风险动作 API 必须能表达“待确认”和“已确认”两个阶段。
- 数据库未定前，业务代码应通过 repository / storage 抽象隔离具体数据库。

## 核心实体

后端必须围绕以下实体建模：

- `Project`
- `Task`
- `Dataset`
- `ModelAsset`
- `RuntimeEnvironment`
- `AiAssistantModel`
- `AssistantSession`
- `Report`

新增实体或核心字段必须先在 OpenSpec Change 中说明，并评估是否需要更新 Global Spec。

## Task 状态机

Task 只允许一个主状态。环境检查、数据检查、模型检查、资源检查、服务启动、训练中、验证评估、导出产物属于 `phase`。

后端必须保证状态流转可追踪，并记录：

- `startedAt`
- `finishedAt`
- `duration`
- `pid`
- `exitCode`
- `errorCode`
- `errorMessage`
- `logRef`

## 任务执行器契约

执行器必须支持：

- 启动任务。
- 停止任务。
- 同步进程状态。
- 采集 stdout / stderr。
- 记录退出码。
- 识别端口冲突、命令不存在、依赖缺失、CUDA OOM 等常见错误。
- 在失败时保留原始日志和结构化诊断结果。

执行器不得直接扫描整盘、静默安装大量依赖或修改全局环境变量。

## 日志规则

- 日志按任务 ID 分片存储。
- 单条日志记录包含 `timestamp`、`stream`、`level`、`message`、`source`。
- API 支持分页读取、关键词过滤和时间排序。
- 日志过大时保留索引和尾部日志，并提示用户导出完整日志。

## 数据、模型和环境检查

数据检查：

- 检查路径、目录结构、标注文件、配置文件、样本统计和异常项。
- 检查失败不应删除已创建记录，应写入错误码和原因。

模型检查：

- 检查模型文件、config、tokenizer、Adapter 关系、格式、大小和资源适配。
- LoRA / Adapter 缺少基础模型时必须标记警告或不可用。

环境检测：

- 检测 Python、包管理器、框架、CUDA、驱动、磁盘、内存。
- 导入本地环境时不要求用户手填框架和 CUDA 版本。
- 依赖安装和修复必须记录日志，并按高风险操作确认。

## 推理服务规则

- 启动前检查模型、环境、端口、推理依赖和资源。
- 启动成功后生成 endpoint 并执行健康检查。
- 服务进程崩溃进入异常状态。
- 用户停止服务后进入已停止。
- 健康检查通过不代表服务已完成，推理服务保持运行中。
