# 0-1 开发规划

## 阶段目标

V1 的目标不是一次性覆盖所有 AI 实验场景，而是先跑通“本地项目接入 -> 环境检查 -> YOLO 训练 -> 结果查看 -> 模型服务 -> AI 助手代操作”的闭环。

优先级排序：

1. 项目、环境、任务、模型文件这些基础实体必须先稳定。
2. 每个核心能力都必须有 UI 入口和 AI 入口。
3. 先支持单机本地工作流，再预留服务器和云端资源入口。
4. 先做 YOLO / PyTorch 高频场景，再扩展大模型训练。

## 推荐技术边界

项目采用前后端分离。

前端建议：

- 桌面壳：Electron
- UI：React + TypeScript
- 状态管理：Zustand 或 TanStack Query
- 图表：ECharts 或 Recharts
- 日志/任务流：WebSocket 或 SSE

后端技术方向：

- 业务后端：Python
- API 服务：Python Web/API 框架，优先选择团队熟悉且生态稳定的方案
- 本地任务执行：Python 后端负责进程管理、任务状态、日志采集、端口检测和系统资源读取
- 数据库：暂未确定，工程初始化前需要完成选型；候选包括 SQLite、PostgreSQL、MySQL
- 数据访问：先定义 repository / storage 抽象，避免业务逻辑绑定具体数据库
- 异步任务：先用受控 worker、任务表和进程管理机制，复杂后再引入队列
- 环境管理：conda / venv / uv 适配层

AI 助手技术方向：

- AI 助手服务：Python
- Intent Router：识别创建项目、检查环境、启动训练、诊断错误等意图
- Tool Registry：把项目、环境、训练、服务操作注册成可调用工具
- Context Provider：读取当前页面、当前项目、最近任务、日志和环境状态
- Execution Plan：AI 每次操作生成步骤，并通过任务面板执行
- Guardrail：涉及文件删除、覆盖、安装依赖、启动服务等动作时需要用户确认
- Python 后端与 Python AI 助手之间通过明确的模块边界、HTTP/RPC 或任务契约通信
- AI 助手不直接任意执行 shell，必须调用 Python 后端暴露的受控工具或经过审计的工具

## 核心数据模型

首版至少需要这些实体：

- Project：项目档案，包含名称、类型、代码目录、数据目录、默认环境、标签、状态
- Dataset：数据集目录、格式、样本数量、校验结果
- RuntimeEnvironment：Python 版本、CUDA、依赖快照、创建方式、可复用状态
- TrainingTask：训练脚本、参数、状态、日志路径、指标、产物目录
- ExperimentRun：指标、配置、权重文件、报告、对比关系
- ModelArtifact：权重文件、来源任务、格式、大小、版本
- ModelService：服务端口、模型路径、推理脚本、状态、访问地址
- AssistantSession：用户指令、识别意图、执行计划、步骤结果、上下文快照

## 里程碑

### M0：仓库与工程骨架

目标：三个人能在统一规范下并行开发。

交付：

- 前端工程初始化
- 后端工程初始化
- AI 助手 Python 服务初始化
- API 契约目录
- 数据库选型记录和迁移机制草案
- 本地开发启动脚本
- Git 分支、commit、PR、spec 流程

验收：

- 新成员能在 30 分钟内启动前后端
- Python 后端和 Python AI 助手服务能本地启动并按约定契约协作
- 提交一个 demo 页面和一个健康检查 API
- CI 至少覆盖 lint 和基础测试

### M1：首页与项目管理闭环

目标：用户能创建或导入项目，看到首页资源和最近项目。

交付：

- 首页布局
- 设备资源摘要
- 新建项目
- 导入项目
- 项目概览
- 代码目录和数据目录绑定
- 项目扫描基础能力
- AI 创建项目入口

验收：

- UI 创建项目可落库
- AI 输入“帮我创建 YOLOv8 项目”能补齐信息并创建项目
- 导入混乱目录时能给出扫描结果和待确认项

### M2：环境配置闭环

目标：能检测本机环境，创建或复用 Python 环境，安装依赖并诊断常见错误。

交付：

- 硬件、Python、CUDA、GPU、Docker、WSL 检测
- 环境列表
- 创建环境
- 安装 `requirements.txt`
- 扫描 import 推测依赖
- 环境快照
- AI 环境诊断入口

验收：

- 环境检测结果可复现、可记录
- 安装失败能保留日志和诊断原因
- AI 能解释 CUDA 不可用、依赖缺失、版本不匹配

### M3：训练任务闭环

目标：跑通 YOLO 或 PyTorch 训练任务。

交付：

- 训练脚本选择
- 数据集选择
- 参数配置
- 启动、停止、重试任务
- 实时日志
- 指标解析
- 权重产物登记
- 训练失败诊断
- AI 启动训练入口

验收：

- 用户能从项目中创建训练任务
- 系统能跟踪状态、日志、指标和产物
- AI 能根据当前项目启动训练，并在缺参时追问

### M4：模型服务闭环

目标：能把训练产物启动为本地服务并测试。

交付：

- 权重选择
- 推理脚本选择
- 服务启动和停止
- 端口检测
- 图片上传测试
- API 地址展示
- 服务日志
- AI 启动服务入口

验收：

- `best.pt` 可被识别为训练产物
- 服务启动后能返回访问地址
- 端口占用、脚本错误、权重缺失能诊断

### M5：实验分析与报告

目标：支持多次训练对比和报告生成。

交付：

- 实验列表
- 指标曲线
- 多实验对比
- 关键指标解释
- 实验报告生成
- AI 调参建议

验收：

- 至少支持 loss、mAP、precision、recall 对比
- 报告能引用配置、指标、日志摘要和产物
- AI 建议能说明依据和风险

## 三人协作分工

建议默认分工：

- A：前端与桌面壳，负责页面、交互、状态、任务面板、AI 助手 UI
- B：Python 后端与本地执行，负责 API、数据库抽象、环境检测、任务执行、日志和服务管理
- C：Python AI 助手与产品规格，负责意图识别、工具注册、上下文读取、spec 维护和验收用例

每个功能仍需要跨端验收，不能只按个人模块完成。

## 风险与决策点

- 桌面壳选型：V1.0 采用 Electron，需要尽早固化主进程、预加载脚本、渲染进程和打包配置边界。
- 数据库选型：当前暂未确定，需要在 M0 阶段明确本地单机、多人协作、迁移成本和查询复杂度要求。
- 后端与 AI 助手边界：Python 后端负责稳定业务能力和本地执行，Python AI 助手负责意图识别、工具编排和模型生态；模块或服务契约必须尽早固化。
- 环境管理边界：自动安装依赖风险高，必须设计确认机制和失败回滚。
- 本地文件权限：扫描目录、启动脚本、删除产物都要有明确权限提示。
- AI 操作可控性：AI 只能生成计划和调用已注册工具，不能直接任意执行 shell。
- 训练任务复杂度：先支持常见模板，再开放自定义脚本。
- 大模型训练成本：V1 只做入口预留和小模型 demo，不把完整训练平台作为首版目标。

## V1 验收标准

- 用户可以从首页进入项目、环境、训练、服务和 AI 助手。
- 项目创建、环境检测、训练启动、模型服务四条主链路可走通。
- 同一功能至少有一个 UI 操作路径和一个 AI 操作路径。
- 所有任务都有状态、日志、失败原因和下一步建议。
- 需求变更有 spec，代码变更有 commit，PR 能对应到变更记录。
