# PR Review Checklist

每个 PR 必须同时接受 Code Review 和 Spec Review。审查目标是确认代码、OpenSpec、测试和风险边界一致。

## PR 描述必须包含

```text
1. 关联的 OpenSpec Change
2. 影响范围
3. 修改了哪些代码模块
4. 是否修改了 Global Spec
5. 是否修改了 Module Spec
6. 是否新增 API
7. 是否新增数据字段
8. 是否新增状态机状态
9. 是否涉及 AI 行为
10. 是否涉及高风险操作
11. 测试说明
12. 回滚方式
```

## 合并到 dev 前必须检查

- 是否违反 Global Spec。
- 是否违反 Module Spec。
- 是否绕过用户确认。
- 是否让 AI 直接执行高风险动作。
- 是否新增未登记状态。
- 是否新增未登记字段。
- 是否新增未定义 API。
- 是否有必要测试。
- Spec 和代码是否同步。
- 是否包含无关重构或大规模格式化。

## OpenSpec 检查

- Change 目录是否存在。
- `proposal.md` 是否说明背景、目标、非目标、影响范围和风险。
- `tasks.md` 是否反映真实完成状态。
- 涉及架构、API、数据模型、状态机或复杂流程时是否有 `design.md`。
- 验收记录是否包含命令、结果和遗留问题。
- 是否说明 UI 入口和 AI 入口。

## 前端检查

- 页面是否服务真实工作流，不是营销式页面。
- 表单是否有 loading、success、failed、cancelled 或等价状态。
- 高风险操作是否有确认弹窗或确认步骤。
- 错误信息是否能被用户理解，并能支持 AI 诊断。
- 页面上下文是否按规则传递给 AI 助手。
- 文本、按钮、列表、日志区域是否在常见窗口尺寸下可用。

## 后端检查

- API 返回结构是否稳定。
- 错误码和错误消息是否可用于 UI 展示和 AI 诊断。
- 文件扫描是否限制范围。
- 长任务是否可追踪、可取消、可查看日志。
- 任务状态机是否只使用已登记状态。
- 依赖安装、服务启动、环境修改是否保留日志并需要确认。

## AI 助手检查

- AI 输出是否区分 `explain`、`field_suggestion`、`action_request`。
- `action_request` 是否只调用后端受控工具。
- 高风险动作是否在真实操作入口中等待用户确认。
- 是否避免展示 API Key、token、环境变量和敏感路径。
- 失败时是否返回结构化原因、日志引用和下一步建议。

## 回滚检查

PR 必须说明：

- 如何回滚代码。
- 是否需要回滚数据字段或状态。
- 是否存在已经写入本机文件、环境或服务配置的副作用。
- 回滚后 Spec 是否需要同步恢复。
