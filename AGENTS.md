# AGENTS.md

本文件是本项目所有 Codex agent 的工作协议。任何使用 Codex 参与本项目开发、重构、测试、文档整理或 PR 修复的任务，都必须先阅读并遵守本文件。

## 项目背景

项目名称：智算桌面

产品定位：面向本地 AI 实验的桌面应用，管理用户的代码、数据、运行环境、训练任务、模型文件、模型服务和实验结果，并用 AI 助手把复杂操作转化为可执行任务。

核心原则：

- 所有核心功能必须同时提供 UI 点击入口和 AI 助手入口。
- AI 助手不是聊天框，而是项目、环境、训练、服务、诊断和分析的操作中枢。
- 项目采用前后端分离，后续可封装为桌面应用。
- 所有功能修改必须有需求记录、代码实现、验证结果和 git commit。

## 重要文档

Agent 开始工作前，应优先查看这些文件：

- `README.md`：项目入口
- `docs/prd/V1.md`：V1 产品需求
- `docs/planning/development-plan.md`：0-1 开发规划
- `docs/process/vibe-coding.md`：vibe coding 协作规范
- `docs/specs/README.md`：需求变更规范
- `docs/specs/changes/`：所有需求变更记录

如果用户请求与这些文档冲突，先指出冲突并说明建议处理方式；除非用户明确要求，否则不要静默覆盖既有规范。

## 工作总原则

1. 先读上下文，再动手。
2. 先确认需求边界，再写代码。
3. 先写或更新 spec，再开发功能。
4. 代码、文档、测试和 git 记录必须能互相追踪。
5. 不扩大需求范围，不顺手做无关重构。
6. 不删除或回滚用户已有修改，除非用户明确要求。
7. 每次完成可交付修改后都要提交 git。

## 标准工作流

每个功能变更按以下流程执行：

1. 检查仓库状态：`git status --short --branch`
2. 阅读相关 PRD、规划、既有 spec 和代码
3. 在 `docs/specs/changes/<change-id>/` 创建或更新变更记录
4. 编写或更新：
   - `proposal.md`
   - `tasks.md`
   - `design.md`，仅在涉及架构、API、数据模型或复杂流程时必需
   - `acceptance.md`
5. 实现代码或文档变更
6. 运行必要验证
7. 更新验收结果和任务清单
8. 提交 git
9. 在最终回复中说明变更文件、验证结果和 commit

纯文档小改也需要 commit。只有拼写、标点、格式这类极小修改可以不新增完整 spec，但仍要保证 git 记录清楚。

## Git 规范

分支建议：

- `main`：稳定主分支
- `develop`：集成分支
- `feature/<change-id>`：功能分支
- `fix/<change-id>`：修复分支
- `docs/<change-id>`：文档分支

提交格式：

```text
<type>(<scope>): <summary>
```

常用类型：

- `feat`：功能
- `fix`：修复
- `docs`：文档
- `spec`：需求规格
- `refactor`：重构
- `test`：测试
- `chore`：工程维护

示例：

```text
spec(project): add import project workflow
feat(project): implement local project scanner
test(project): cover scanner edge cases
docs(agent): add codex development guide
```

禁止：

- 把多个无关功能塞进一个 commit
- 用 `git reset --hard` 或强制回滚用户修改
- 提交未解释的大规模格式化
- 跳过验证后声称已完成

## 需求与 Spec 规则

新增、修改或删除功能时，必须先维护 spec。

Spec 目录：

```text
docs/specs/changes/<change-id>/
  proposal.md
  tasks.md
  design.md
  acceptance.md
```

每个 proposal 至少说明：

- 背景
- 目标
- 非目标
- UI 入口
- AI 入口
- 需求场景
- 影响范围
- 风险

每个 acceptance 至少记录：

- 验收环境
- 验收用例
- 测试命令或人工验证结果
- 遗留问题

## 产品约束

实现任何业务能力时，必须同时考虑 UI 入口和 AI 入口。

示例：

- 新建项目：既要有页面表单，也要有 AI 创建项目工具。
- 环境检测：既要有环境配置页面，也要支持 AI 回答“为什么 CUDA 不可用”。
- 启动训练：既要有任务创建页面，也要支持 AI 根据当前项目补齐参数并启动。
- 模型服务：既要能从模型文件页面启动，也要支持 AI 识别最新 `best.pt` 并启动服务。

如果当前迭代只实现其中一个入口，必须在 spec 中明确另一个入口的补齐任务和验收标准。

## 推荐架构方向

前端：

- React + TypeScript
- 组件和页面分层清晰
- 所有任务型操作要有 loading、success、failed、cancelled 状态
- 任务执行面板必须展示步骤、日志、失败原因和下一步建议

后端：

- Python FastAPI
- SQLite 起步
- 本地任务通过受控 worker 或子进程管理
- 所有文件扫描、依赖安装、训练启动、服务启动都要有日志
- 不允许让 AI 直接执行任意 shell，必须通过受控工具层

AI 编排：

- Intent Router：识别用户意图
- Tool Registry：注册可执行能力
- Context Provider：读取当前页面、项目、环境、任务和日志
- Execution Plan：生成可展示、可审计的步骤
- Guardrail：高风险操作需要用户确认

## 安全边界

必须让用户确认的操作：

- 删除文件、数据集、模型权重或训练产物
- 覆盖已有配置或环境
- 安装、卸载或升级大量依赖
- 修改系统级 Python、CUDA、Docker、WSL 配置
- 启动长时间训练任务
- 暴露本地服务端口
- 上传用户数据到外部服务

默认禁止：

- 直接递归删除目录
- 静默修改全局环境变量
- 静默联网下载大文件
- 将本地路径、日志、数据集内容发送到外部服务，除非用户确认

## 前端实现要求

- 页面应服务真实工作流，不做营销式落地页。
- 工作台类页面优先信息密度、可扫描性和稳定布局。
- 对任务、环境、训练、服务这类状态型功能，要提供清晰的状态标签和错误信息。
- 所有按钮、表单、列表、空状态、失败状态都要可用。
- 复杂操作需要步骤条或任务面板，不只弹 toast。
- 图表和日志区域要考虑大数据量和长文本。

## 后端实现要求

- API 返回结构必须稳定，错误码和错误消息要可用于 UI 展示和 AI 诊断。
- 本地路径、环境、训练任务、服务进程都要持久化必要状态。
- 长任务必须可追踪、可取消、可查看日志。
- 环境检测和依赖安装要保留原始日志与结构化诊断结果。
- 文件扫描必须限制范围，避免无边界扫描整盘。

## AI 助手实现要求

AI 助手的输出不能只是一段文字。涉及操作时必须形成结构化执行计划：

```text
任务：创建 YOLOv8 项目
步骤：
1. 检查代码目录
2. 检查数据集目录
3. 检测 Python 和 CUDA
4. 推荐运行环境
5. 创建项目档案
6. 给出下一步建议
```

AI 工具调用必须满足：

- 输入参数明确
- 输出结果结构化
- 失败原因可解释
- 日志可追踪
- 高风险步骤可中断或确认

## 测试与验收

根据变更风险选择验证强度。

最低要求：

- 文档变更：检查链接、路径和格式
- 前端变更：运行 lint/typecheck，必要时用浏览器检查关键页面
- 后端变更：运行单元测试或接口测试
- AI 编排变更：至少覆盖成功、缺参、失败三类场景
- 数据模型变更：验证迁移和回滚路径

最终回复必须说明哪些验证已运行，哪些没有运行以及原因。

## 三人协作方式

默认角色：

- 前端负责人：页面、交互、状态管理、任务面板、AI 助手 UI
- 后端负责人：API、数据库、本地环境、训练任务、日志和服务管理
- AI/产品负责人：PRD、spec、AI 工具编排、上下文、验收用例

协作规则：

- 公共 API、数据模型、任务状态枚举变更前，先更新 design 或接口文档。
- 前后端可以基于 mock 并行，但最终必须用真实 API 联调。
- 每天结束前提交可解释的 git commit，避免长期保留散乱本地修改。
- 任何人发现 spec 与实现不一致，应先修正文档或提出变更，不要让差异继续扩大。

## Agent 回复规范

Agent 完成任务后的回复应包含：

- 修改了什么
- 关键文件路径
- 验证结果
- git commit
- 仍需团队确认的事项

回复要简洁，不要长篇复述代码。
