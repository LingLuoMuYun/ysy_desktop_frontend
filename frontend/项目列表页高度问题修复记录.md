# 项目列表页高度问题修复记录

## 问题描述

项目列表页（`ProjectsPage`）在只有一个项目卡片时，`project-list-shell` 高度极低（仅 ~192px），未能填满页面剩余空间。

## 完整排查过程

### 第一轮：移除冗余的 `height: 100%`

**怀疑点**：`.project-list-shell` 和 `.project-grid` 同时设置了 `height: 100%` 和 `flex: 1 1 auto`，两者在 flex column 布局中可能冲突——`height: 100%` 的百分比解析可能产生循环依赖，导致浏览器回退到内容高度。

**改动**：从两处各删掉 `height: 100%`，只保留 `flex: 1 1 auto`。

**结果**：❌ 无效，高度仍然很低。

### 第二轮：改用 `flex: 1` 替代 `flex: 1 1 auto`

**怀疑点**：`flex: 1 1 auto` 的 `flex-basis: auto` 会先按内容大小（~150px）计算初始尺寸，可能导致剩余空间分配异常。

**改动**：将 `.project-list-shell` 和 `.project-grid` 的 `flex: 1 1 auto` 改为 `flex: 1`（即 `flex: 1 1 0%`），让元素从 0 开始分配空间。

**结果**：❌ 无效，高度仍然很低。

### 第三轮：逐层排查父链高度

**方法**：通过浏览器 DevTools 的 Computed 面板，从顶向下逐层查看元素的实际 computed height：

| 元素 | computed height | 状态 |
|------|----------------|------|
| `.project-page` | 771px | ✅ 正常 |
| `.project-list-shell` | 192px | ❌ 仅内容高度 |

**结论**：父容器 `.project-page` 高度正常（771px），说明问题不在高度传递链，而在 `.project-page` 内部的布局模式。

### 第四轮：定位根因——CSS 特异度冲突

**怀疑点**：`.project-page` 的 `display: flex` 是否被 `.workbench-page` 的 `display: grid` 覆盖？

**验证**：DevTools Computed 面板中查看 `.project-page` 的 `display` 值 → **确认为 `grid`**，而非预期的 `flex`。

**根因分析**：

```css
/* globals.css — 后加载，优先级更高 */
.workbench-page {
  display: grid;                    /* ← 这条赢了！ */
  grid-template-rows: 42px minmax(0, 1fr);
  height: 100%;
  overflow: hidden;
}

/* ProjectsPage.css — 先加载，被覆盖 */
.project-page {
  display: flex;                    /* ← 被覆盖，未生效 */
  flex-direction: column;
  gap: 14px;
  height: 100%;
}
```

两者特异度相同（0-1-0），Vite 打包时 `globals.css` 排在 `ProjectsPage.css` 后面，`.workbench-page` 胜出。

由于 `display: grid` + `grid-template-rows: 42px minmax(0, 1fr)` 只有 **2 个显式行**，而页面有 3 个子元素：

| 行 | 类型 | 高度 | 内容 |
|----|------|------|------|
| 行 1 | 显式 | 42px | `.project-page__toolbar` |
| 行 2 | 显式 | 1fr（剩余空间） | `.project-search` |
| 行 3 | **隐式（auto）** | **192px（内容大小）** | `.project-list-shell` |

`.project-list-shell` 被挤到隐式第 3 行，`grid-auto-rows: auto`，仅获得内容高度。

## 最终修复

**文件**：`frontend/src/pages/ProjectsPage.css`

**改动汇总**：

### 1. 修复特异度冲突（关键修复）

```css
/* 改前 — 特异度 0-1-0，被 globals.css 覆盖 */
.project-page {
  display: flex;
  ...
}

/* 改后 — 特异度 0-2-0，确保赢过 .workbench-page */
.workbench-page.project-page {
  display: flex;
  flex-direction: column;
  gap: 14px;
  padding: 14px 24px 14px 24px;
  background: var(--surface);
  height: 100%;
  min-height: 0;
  overflow: hidden;
}
```

### 2. 清理冗余属性

```css
/* .project-list-shell — 移除冗余的 height: 100%，flex: 1 1 auto 改为 flex: 1 */
.project-list-shell {
  position: relative;
  z-index: 0;
  display: flex;
  flex-direction: column;
  flex: 1;           /* 原为 flex: 1 1 auto */
  min-height: 0;
  /* 移除: height: 100% */
  overflow: hidden;
  border: 1px solid var(--line);
  border-radius: var(--radius);
}

/* .project-grid — 同上 */
.project-page .project-grid {
  display: flex;
  flex: 1;           /* 原为 flex: 1 1 auto */
  flex-direction: column;
  align-items: stretch;
  justify-content: flex-start;
  gap: 12px;
  min-height: 0;
  /* 移除: height: 100% */
  padding: 12px 14px 14px;
  background: var(--surface);
}
```

## 经验教训

1. **CSS 类名冲突排查**：当元素同时拥有多个 class，且不同 CSS 文件设置了冲突属性时，排查顺序应该是：特异度 → 源码顺序 → 打包顺序。DevTools Computed 面板是定位此类问题的最快手段。

2. **`height: 100%` 与 `flex: 1` 不应同时使用**：在 flex column 布局中，`flex: 1`（或 `flex: 1 1 auto`）已经负责让子元素填满剩余空间，额外加 `height: 100%` 是冗余的，且可能在某些浏览器中引发百分比解析的循环依赖。

3. **CSS 特异度管理**：页面级覆盖基类样式时，应该用 `.base.page` 双类选择器（特异度 0-2-0）而非单类选择器，确保覆盖一定生效，不依赖 CSS 文件加载顺序。

4. **逐层诊断法**：遇到布局问题，从目标元素向父链逐层检查 computed height/width，能快速定位高度/宽度是在哪一层"断掉"的。
